# ABOUT

## Что это за проект

`vue-oneblock-carousel` - небольшой проект на `Vue 3 + Vite + TypeScript`, в котором основной объект разработки это компонент карусели `VObCar`.

Сейчас репозиторий решает сразу две задачи:

1. Хранит сам переиспользуемый UI-компонент карусели.
2. Даёт локальную demo-страницу с несколькими примерами использования, чтобы быстро проверить внешний вид, слоты и пустое состояние.

Проект не выглядит как полноценная npm-библиотека с отдельным public API и сборкой пакета. По текущей структуре это скорее изолированная витрина/песочница компонента с unit-тестами.

## Основной стек

- `Vue 3`
- `Vite`
- `TypeScript`
- `SCSS`
- `Vitest`
- `@vue/test-utils`
- `ESLint`

Дополнительно используется только `normalize.css`. Новые зависимости без реальной необходимости добавлять не стоит.

## Главная цель проекта

Центр проекта - компонент [`src/components/VObCar.vue`](src/components/VObCar.vue), который:

- показывает один активный слайд;
- умеет переключать слайды по точкам;
- поддерживает автопрокрутку по таймеру;
- поддерживает свайпы и drag-жесты для touch-устройств;
- поддерживает обычное переключение и режим направленной анимации `oneway`;
- позволяет глубоко кастомизировать разметку через scoped slots;
- поддерживает пустое состояние через слот `not-found`.

## Как устроен `VObCar`

### Входные данные

Компонент принимает props:

```ts
type SlideId = number | string

type CarSlide = {
  id?: SlideId
  cover: string
  title: string
  text: string
} & Record<string, unknown>

defineProps<{
  animationType?: "" | "oneway"
  autoplay?: boolean
  autoplayInterval?: number
  slides: CarSlide[]
}>()
```

Обязательные поля слайда для дефолтной разметки:

- `cover` - URL изображения;
- `title` - заголовок;
- `text` - описание;
- `id` - необязателен, если не передан, используется индекс.

Слайд специально расширен через `Record<string, unknown>`, чтобы в кастомных слотах можно было использовать произвольные дополнительные поля.

Дополнительные props поведения:

- `autoplay` - включает автоматическое перелистывание слайдов;
- `autoplayInterval` - задаёт интервал автопрокрутки в миллисекундах.

### Поддерживаемые слоты

Компонент поддерживает два уровня кастомизации:

1. Полная замена всего слайда:
   - `slide`
2. Частичная замена внутренних частей:
   - `image`
   - `content`
   - `title`
   - `text`
   - `dots`
   - `dot`
   - `not-found`

Через слот `slide` можно полностью переопределить порядок блоков и логику навигации, при этом компонент пробрасывает полезные scoped props:

- `slide`
- `index`
- `slides`
- `activeIndex`
- `activeDotIndex`
- `goToSlide(index)`
- `highlighterStyle`
- `setDotElement`
- `getSlideKey`

Это важно: даже при глубокой кастомизации текущая архитектура ожидает, что навигация по точкам при необходимости сможет использовать `setDotElement`, иначе highlighter не сможет корректно считать размеры и позицию.

## Внутренняя логика состояния

Внутри компонента есть несколько ключевых сущностей:

- `activeIndex` - реально показанный слайд;
- `targetIndex` - индекс, в который идёт анимационный переход;
- `activeDotIndex` - индекс активной точки во время перехода;
- `animClass` - CSS-класс направления анимации;
- `dotElements` - ссылки на DOM-кнопки точек;
- `highlighter` - reactive-стиль подложки активной точки;
- `autoplayTimeoutId` и `isHovered` - состояние автопрокрутки и её паузы;
- drag-состояние указателя - стартовые координаты, текущее смещение и флаг активного жеста.

Логика переключения построена на таймерах:

1. При клике вызывается `goToSlide(index)`.
2. Сбрасываются старые таймеры.
3. Высчитываются метрики прошлой и следующей точки.
4. Highlighter сначала растягивается между точками.
5. Через `150ms` компонент переходит в промежуточное состояние `stop`.
6. Через `300ms` фиксируется новый `activeIndex`, и анимационный класс очищается.

Это не CSS-only решение: здесь есть явная JS-координация состояния и анимации.

## Автопрокрутка

Компонент умеет автоматически переключать слайды по таймеру.

Текущее поведение:

- автопрокрутка включается через `autoplay`;
- по умолчанию используется интервал `5000ms`, либо значение из `autoplayInterval`;
- автопрокрутка работает только если слайдов больше одного и нет активного перехода;
- при `mouseenter` таймер ставится на паузу;
- после ручного переключения таймер запускается заново;
- для пустого и одиночного набора слайдов таймер не стартует.

## Свайпы и drag-жесты

Компонент поддерживает горизонтальные жесты через `pointer`-события.

Текущее поведение жестов:

- свайп влево переключает на следующий слайд;
- свайп вправо переключает на предыдущий;
- короткий drag ниже порога игнорируется;
- жест игнорируется, если движение похоже на вертикальный скролл страницы;
- во время drag автопрокрутка ставится на паузу;
- если переход уже идёт, новый жест не запускает дополнительную анимацию;
- циклического свайпа на краях пока нет.

Это намеренно простой сценарий: компонент не тянет контент за пальцем в реальном времени, а лишь определяет направление жеста и использует существующий `goToSlide`.

## Как работает анимация

Поддерживается два режима:

- `""` - без направленного класса, только смена состояния и highlighter;
- `"oneway"` - при переходе добавляется либо `animated-right`, либо `animated-left`.

В режиме `oneway` изображение и текст уезжают в сторону, затем получают состояние `stop`, после чего рендерится новый активный слайд.

Визуальная часть анимации зашита в стилях самого компонента, а highlighter двигается через inline style из reactive-состояния.

## Пустое состояние

Если `slides.length === 0`, компонент не рендерит `.v-ob-car-slide` и показывает слот `not-found`.

Это уже покрыто тестом и считается частью публичного поведения.

## Demo-страница

[`src/App.vue`](src/App.vue) не просто монтирует компонент, а служит живой документацией.

Сейчас на странице есть:

- базовая карусель с `animation-type="oneway"`;
- карусель с автопрокруткой и паузой при наведении;
- пример кастомизации `title` и `dot` через слоты;
- пример пустого состояния через слот `not-found`;
- отдельный блок с идеями дальнейшего развития компонента.

Если нужно быстро понять возможности проекта без чтения тестов, начинать стоит с `App.vue`.

## Тестовое покрытие

Основное тестирование сосредоточено в [`tests/unit/VObCar.spec.ts`](tests/unit/VObCar.spec.ts).

Тесты проверяют:

- рендер первого слайда по умолчанию;
- переход вправо и влево в режиме `oneway`;
- игнорирование клика по уже активной точке;
- работу `slides` из props;
- автопрокрутку, её паузу при hover и перезапуск после ручного переключения;
- свайпы влево и вправо;
- игнорирование короткого drag и вертикального жеста;
- замену частей разметки через слот `title`;
- полную кастомизацию через слот `slide`;
- пустое состояние через `not-found`.

В [`vitest.setup.ts`](vitest.setup.ts) замоканы `offsetLeft` и `offsetTop`, чтобы анимация highlighter была предсказуемой в `jsdom`. Это важная часть тестовой модели, её нельзя ломать без пересмотра самих тестов.

## Ограничения текущей реализации

На данный момент в компоненте нет:

- управления с клавиатуры;

Это не баги сами по себе, а текущие границы реализации.

## Правила для будущей модели

При работе с этим проектом полезно держать в голове следующие принципы:

- Не усложнять архитектуру без необходимости. Проект маленький, здесь важнее простота, чем универсальность.
- Центральная сущность - `VObCar`. Большинство изменений должны либо улучшать его API, либо расширять demo/test coverage.
- Сначала проверять, можно ли решить задачу существующими props, slot API и локальным состоянием, а не вводить новый слой абстракции.
- Следовать текущему стилю `Vue 3`: `script setup`, Composition API, локальная типизация, ранние возвраты, минимум магии.
- Не добавлять библиотеки ради анимации, слайдера или состояния, пока задачу реально можно решить встроенными средствами Vue и текущим CSS/JS.
- При изменении публичного поведения компонента обновлять и demo-страницу, и unit-тесты.
- При кастомизации слотов помнить, что дефолтная разметка и CSS-классы уже участвуют в тестах и пользовательских примерах.

## Короткий вывод

Это компактный репозиторий с одной основной задачей: развивать и демонстрировать компонент карусели `VObCar`.

Если модель должна вносить изменения, её безопасная стратегия такая:

1. Сначала понять, относится ли задача к самому компоненту, demo-странице или тестам.
2. Потом сохранить простую структуру без лишних слоёв.
3. После любого заметного изменения синхронизировать поведение компонента, примеры в `App.vue` и тесты в `VObCar.spec.ts`.
